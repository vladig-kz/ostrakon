# Ostrakon

Telegram-бот для **демократического бана спамеров** в группах: участники голосуют за
кик, бот применяет результат. Чистый PHP, без фреймворков, рассчитан на обычный
shared-хостинг.

- Полная спецификация: [`SPEC.md`](SPEC.md)
- Формулы рейтинга («аксакал»): [`FORMULAS.md`](FORMULAS.md)
- English README: [`README.md`](README.md)

> **Статус: функционально готов, ждёт боевого тестирования.** Готово: инфраструктура,
> асинхронный вебхук, голосования (режимы light и full с рейтингом активности и статусом
> «аксакал»), многоязычная веб-панель с персональным контролем доступа, личные уведомления
> в ЛС (с командами-ответами), все cron-задачи и экспорт/импорт JSON (`Exporter`) для
> переноса группы между инстансами.

---

## Как это работает

- **В группе бот делает ровно одно: запускает голосование.** Участник отвечает (reply) на
  сообщение нарушителя, написав в ответе **только** упоминание бота (`@yourbot` и ничего
  больше) — бот публикует голосование с кнопками «За / Против». (Требование «только упоминание»
  не даёт объясняющей фразе, где просто есть `@yourbot`, случайно вынести кого-то на
  голосование; если бота упомянули иначе — он просто отвечает короткой подсказкой.) **Команд, набираемых в группе, нет** (не засоряем её; безопасно,
  когда в чате несколько ботов — упоминаешь конкретного). Команды у бота есть только в ЛС:
  `/start`, `/groups`, `/language` и `/help`, плюс модерация ответом на уведомление — см. раздел про панель.
- **Всё остальное — из веб-панели** (`/admin`, вход через Telegram): защита участников,
  разбан, настройки группы, журнал голосований, подбор параметров «аксакала».
- **Итог голосования.** Голоса суммируются по *весу*. Достигнут порог бана — участник
  кикнут; достигнут порог отклонения — голосование закрыто. **Админ** решает мгновенно
  нажатием кнопки. Таймауты (T1/T2) закрывают зависшие голосования.
- **Два режима на группу:**
  - **light** — только голосования, без учёта сообщений.
  - **full** — дополнительно считает активность и вычисляет затухающий **score**;
    участники со `score ≥ elder_threshold` становятся **«аксакалами»**: их голос весит
    больше, и рядом с именем появляется видимый тег.

## Приватность — что храним и что нет

Бот намеренно минимален по данным:

- **Храним:** `user_id`, `@username` (если есть), время входа, агрегаты по участнику
  (`score`, **число** сообщений), флаги бана, настройки группы, голосования (с текстом
  *сообщения-триггера* — для журнала модерации) и небольшие пользовательские настройки
  (выбранный язык ЛС, персональные подписки админов на уведомления).
- **Не храним:** текст обычных сообщений и имена/фамилии пользователей. В режиме full
  пишутся только **метаданные** сообщения (кто, когда, на что ответ) — не содержимое.

## Требования к хостингу

| Компонент      | Требование                                                                                     |
|----------------|------------------------------------------------------------------------------------------------|
| PHP            | **8.1+** (разработка на 8.2)                                                                   |
| Расширения PHP | `pdo_mysql`, `curl`, `json` (обязательно); `mbstring` (желательно)                             |
| СУБД           | MySQL 5.7+ **или** MariaDB 10.x (InnoDB, utf8mb4)                                              |
| Веб-сервер     | Apache (`.htaccess` в комплекте) или Nginx (каталоги закрыть вручную) + `mod_rewrite`          |
| HTTPS          | **Обязателен** — вебхук и Login Widget работают только по HTTPS                                |
| Cron           | Возможность запускать `cron.php` по расписанию (раз в минуту — идеально; реже — тоже работает) |

`fastcgi_finish_request` **не требуется**: вебхук асинхронный (кладёт апдейт в очередь и
сразу отдаёт `200`, а очередь разбирает cron-воркер — см. [Асинхронный вебхук](#асинхронный-вебхук)).

> **Единый источник времени — база данных.** Все метки времени хранятся и сравниваются в UTC
> по часам сервера БД (читаются один раз на соединение). PHP и БД могут стоять на разных
> хостах с разошедшимися часами — на корректность это не влияет.

Размещение задаётся через `APP_URL` и поддерживает оба варианта:
- поддомен: `https://ostrakon.example.com`
- подпапка: `https://example.com/ostrakon`

## Настройка бота (@BotFather)

### 1. Создать бота
`/newbot` → задать имя и username, оканчивающийся на `bot`. BotFather выдаст **токен** →
в `config/bot.php` (`BOT_TOKEN`).

### 2. Настроить бота
- **Privacy-режим — для режима `full` его НУЖНО отключить:** `/setprivacy` → выбрать бота →
  **Disable**. При включённом privacy бот не видит обычные сообщения, и активность/score
  не посчитать. (Инициирование голосования через `@упоминание` работает в любом случае.)
  Смена privacy применяется только при **перевходе** бота: после Disable удалите бота из
  группы и добавьте заново.
- **Домен для Login Widget** (для панели): `/setdomain` → выбрать бота → хост из `APP_URL`
  (например `ostrakon.example.com`, только хост — без `https://` и пути).
- **Аватар бота** (необязательно): `/setuserpic` → выбрать бота → загрузить квадратную
  картинку (Telegram обрежет в круг). Можно взять `assets/favicon.png` (512×512). Там же
  задаются имя, `/setdescription` и `/setabouttext`.

### 3. Права бота в группе
Добавить бота **администратором** (админство обязательно, иначе не приходят события
входа/выхода и нельзя банить). Включить **только** эти права, остальное выключить:
- **Блокировка пользователей** — бан и readonly во время голосования.
- **Удаление сообщений** — удаление сообщения-спама/активации.
- **Управление тегами участников** (`can_manage_tags`) — показ тега «аксакал» в режиме
  full. Необязательно, если теги аксакала не используются.

### 4. Webhook
`install.php` регистрирует его автоматически из `APP_URL` + `WEBHOOK_SECRET`. Вручную
(если нужно):

```bash
curl -F "url=https://ВАШ_ДОМЕН/webhook.php" \
     -F "secret_token=ВАШ_WEBHOOK_SECRET" \
     -F 'allowed_updates=["message","callback_query","chat_member","my_chat_member"]' \
     "https://api.telegram.org/botВАШ_ТОКЕН/setWebhook"
```

> `allowed_updates` обязан включать `chat_member` и `my_chat_member`, иначе бот не увидит
> входы/выходы/повышения. Установщик передаёт правильный набор.

## Развёртывание

1. **Залить** проект в каталог, соответствующий `APP_URL`.
2. **Создать** БД (utf8mb4) и пользователя.
3. **Создать конфиги** из шаблонов и заполнить:
   - `cp config/bot.example.php config/bot.php` — `BOT_TOKEN`, `BOT_USERNAME`,
     `WEBHOOK_SECRET`, `APP_URL`, `SUPERADMIN_TOKEN`, `LOG_LEVEL`.
   - `cp config/db.example.php config/db.php` — доступ к БД, `DB_TABLE_PREFIX`.
4. **Настроить бота** у @BotFather (шаги 1–3 выше).
5. **Запустить установщик** — проверит требования, применит **миграции**, зарегистрирует webhook
   и меню команд бота в ЛС (`/start`, `/groups`, `/language`, `/help` — вручную в @BotFather
   задавать не нужно):
   - Веб: `https://ВАШ_ДОМЕН/install.php?token=ВАШ_SUPERADMIN_TOKEN`
   - CLI: `php install.php`
   - Пропустить регистрацию webhook: `?webhook=0` (веб) / `php install.php nowebhook` (CLI).
6. **⚠️ Удалить `install.php`** после успешной установки.
7. **Настроить системный cron** на `cron.php` (идеально — раз в минуту):
   - CLI: `* * * * * /usr/bin/php /полный/путь/к/cron.php >/dev/null 2>&1`
   - или по URL: `https://ВАШ_ДОМЕН/cron.php?token=ВАШ_SUPERADMIN_TOKEN`
8. Убедиться, что `logs/` доступен веб-серверу на запись.
9. (Опционально) Убрать dev-инструменты `log.php` и `inspect.php` в проде.

Установщик идемпотентен (`CREATE TABLE IF NOT EXISTS`, `INSERT IGNORE`, учёт миграций) —
повторный запуск безопасен.

(`history_days`, тайминг воркера) и интервалы cron. Подробности — в комментариях файла.

## Веб-панель администратора

Открыть `APP_URL/admin` и **войти через Telegram** (Login Widget; подпись проверяется по
`BOT_TOKEN`). Пароли нигде не вводятся. Разделы:

- **Мои группы** — группы, где вы админ и бот присутствует.
- **Участники** — поиск / мультисортировка / пагинация; protect / unprotect / unban;
  в режиме full колонка «% аксакальства» и бейджи админ/владелец. Здесь же владелец может
  выдать/забрать право «менеджер» конкретному админу — см. контроль доступа ниже.
- **Настройки** — все настройки группы (режим, пороги, таймауты, параметры аксакала и
  т.д.), сгруппированы и провалидированы. **Только для менеджеров** (владелец или админ с правом).
- **Журнал** — история голосований с фильтром по статусу и поиском.
- **Симулятор** (режим full, **только для менеджеров**) — по реальной активности группы
  подбирает `elder_threshold`, при котором заданная доля активных станет аксакалами за срок.
- **Миграция** (**только для менеджеров**) — экспорт всей группы в JSON-файл и импорт его в
  ту же группу на другом инстансе. См. «Перенос между инстансами» ниже.

**Контроль доступа.** Любой админ группы может protect/unprotect/unban и читать журнал.
Менять настройки, пользоваться симулятором и делать миграцию может только **владелец**
(создатель) — и те админы, кому владелец явно выдал право *менеджера* (хранится по группе).
Другие админы это право выдать не могут. При снятии админки право теряется автоматически (оно
действует, только пока пользователь админ). Выдача/забор — из списка участников, только владельцу.

Кроме того, подключить бота может только **владелец** группы: Telegram часто разрешает добавлять
ботов и обычным участникам, поэтому при первом добавлении бот проверяет, что добавивший —
создатель; иначе присылает короткое уведомление и выходит. При корректном первом добавлении бот
постит короткую самоудаляющуюся подсказку (исчезает через ~10 мин) с просьбой к владельцу:
(1) открыть с ботом диалог для настройки и (2) назначить его администратором с правом банить
участников — без этого права бот не сможет исполнить итог голосования.

Каждое изменяющее действие серверно перепроверяет, что вы админ *именно этой* группы, а
POST-запросы защищены CSRF.

**Язык интерфейса.** В шапке панели — плоский переключатель (по ссылке на каждый доступный
язык, виден до и после входа). Для **залогиненного** пользователя выбор хранится в базе
(`users.lang`) и является единым источником истины — он общий с диалогом бота: смена языка на
сайте или командой `/language` в ЛС меняет и там, и там (каждая смена со временем; побеждает
самая свежая). Кука `ostrakon_lang` несёт лишь выбор, сделанный на сайте до входа (или из другого
браузера); при входе она сверяется с сохранённым значением по времени. Пока выбора нет — берётся
`Accept-Language` браузера, затем язык по умолчанию (первый файл языка). Чтобы добавить язык,
положите файл `lang/NN-<код>.php` с теми же ключами (`_language_name` / `_language_code` внутри
задают отображаемое имя и код; `NN` в имени файла задаёт порядок — первый файл является языком по
умолчанию).

**Персональные уведомления.** Любой админ может включить себе (по каждой группе, на странице
группы) личные оповещения от бота: старт голосования (кто и против кого), исход (бан / отклонено /
истекло / отменено) и появление/исчезновение аксакалов. По умолчанию выключено, и каждый управляет
только своими — владелец за других включить не может. Доставка в личку, поэтому работает, только
если у админа открыт диалог с ботом; при включении панель шлёт тестовый DM и, если не достучалась,
предлагает открыть диалог. Язык уведомлений — свой у каждого: выбирается клавиатурой на первом
`/start` (или позже командой `/language`), а также запоминается из панели при постановке галочки.
Уведомления о статусе аксакала приходят, даже если видимый тег отключён.

**Быстрые действия из личного диалога.** Админ может модерировать, *отвечая* на уведомление (не
открывая панель и не засоряя группу). На уведомление о старте голосования — ответ `forceban`,
`cancelban` или `protect` (отменить голосование и защитить); на уведомление о бане — `unban` или
`protect`. Бот проверяет, что вы всё ещё админ этой группы и что голосование ещё идёт, затем
выполняет действие и подтверждает.

## Режимы: light и full

- **light** — бот только запускает голосования. Сообщения не учитываются; privacy можно
  оставить включённым.
- **full** — дополнительно пишет **метаданные** сообщений (не содержимое), прошедших
  фильтры качества (`min_msg_length`, `msg_cooldown_minutes`), и периодически пересчитывает
  затухающий **score** (`score_recalc`). Участники со `score ≥ elder_threshold` становятся
  **аксакалами**: вес их голоса — `elder_weight`, и (если задан `elder_title`) им ставится
  видимый тег, автоматически появляющийся/исчезающий при пересечении порога. Требует
  privacy OFF и права `can_manage_tags`. Затухание score — по `FORMULAS.md`
  (`score = Σ exp(−λ·возраст)`, `λ = ln2 / halflife_days`).

  **Ручное назначение аксакалом** (список участников, любой админ, режим full): делает
  участника аксакалом *сейчас*, подкладывая фейковую историю сообщений (по текущим настройкам
  группы), чей затухающий score достигает `elder_threshold`. Score остаётся честным — переживает
  пересчёт и дальше **затухает**, так что удержание статуса зависит от реальной активности.
  Отмены нет.

## Асинхронный вебхук

Т.к. на shared-хостинге часто нет `fastcgi_finish_request`, апдейты никогда не
обрабатываются в самом запросе вебхука:

1. `webhook.php` проверяет secret, кладёт сырой апдейт в `update_queue` (дедуп по
   `update_id`), сразу отдаёт `200` и best-effort «пинает» воркер коротким self-запросом.
2. `cron.php` — **воркер**: держит `flock` (без наложения запусков), крутится
   `worker_loop_seconds` и каждые пару секунд разбирает очередь (`Handler::handle` на
   апдейт) и выполняет наступившие задачи.

Так ответ быстрый независимо от `fastcgi_finish_request`, и даже редкий системный cron не
мешает (self-пинок закрывает промежутки, когда срабатывает).

## Планировщик задач (cron)

`cron.php` читает таблицу `cron_schedule`: задачи с `next_run_at <= NOW()` выполняются, и
их `next_run_at` сдвигается на интервал из `config/defaults.php`. Так задача срабатывает
вовремя и ровно один раз, даже если системный cron дёргается реже раза в минуту.

Задачи: `vote_timeouts`, `pending_setup_ttl`, `bot_messages_cleanup` (раз в минуту),
`reentry_check` (5 мин), `score_recalc`, `data_ttl` (раз в сутки). От наложения защищает
`flock` на `logs/do_not_delete_this.lock` (снимается автоматически; удалять не нужно).

## Миграции схемы

Изменения схемы после установки — папками `migrations/NNN_описание/` (одно действие на
файл `*.sql`/`*.php`). Папки применяются по возрастанию имени; папка засчитывается, только
если все её файлы прошли без ошибок (учёт в `{prefix}migrations`). В `.sql` используйте
`{prefix}` для имён таблиц.

- Веб: `https://ВАШ_ДОМЕН/migrations/run.php?token=ВАШ_SUPERADMIN_TOKEN`
- CLI: `php migrations/run.php`

## Логирование и dev-инструменты

`logs/app.log` (время в UTC). Уровень — `LOG_LEVEL`:
`FATAL(60) > ERROR(50) > WARNING(40) > INFO(30) > DEBUG(20) > TRACE(10)` — пишутся
сообщения важностью ≥ порога. При отладке — `debug`/`trace`, в проде — `warning`/`error`.

Только для разработки (в проде можно убрать): `log.php` (просмотр/очистка лога) и
`inspect.php` (инспектор/правилка БД). Оба закрыты `SUPERADMIN_TOKEN` → cookie.

## Перенос между инстансами

Группу можно перенести на другой инстанс бота (свой хостинг или уход с публичного) без потери
истории. `chat_id` **не** ремапится — это та же группа Telegram, просто её обслуживает новый бот.

1. Владелец открывает **Миграцию** в панели и скачивает JSON-экспорт группы (настройки,
   участники, статистика сообщений, история завершённых голосований). Токен бота и **активные**
   голосования не выгружаются — активные продолжают идти там, где начаты.
2. Владелец добавляет нового бота в ту же группу (в любом режиме), чтобы голосования уже работали.
3. Владелец загружает экспорт на **Миграции** нового инстанса. Импорт **аддитивный**: история
   вливается рядом с тем, что новый бот уже накопил, а сортировка по датам ставит старые записи
   на своё место. В режиме full рейтинг и статус аксакалов пересчитываются сразу после импорта,
   так что имеющиеся аксакалы видны немедленно (а не только после ближайшего суточного пересчёта).

Импорт идемпотентен (можно повторять): `groups`/`participants` — upsert по естественному ключу;
`messages` и `votes` дедуплицируются по датам, так что повторная загрузка не задвоит историю. Оба
инстанса должны быть на **одной версии формата экспорта** (авто-миграции формата нет) — иначе
импорт отклоняется с понятным сообщением.
