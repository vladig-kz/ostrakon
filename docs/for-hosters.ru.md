# Ostrakon — для хостера бота

Установка и эксплуатация инстанса: требования, настройка у @BotFather, развёртывание, конфиги,
безопасность и зона оператора. Что бот умеет со стороны групп — в
[Для владельцев и админов](for-admins.ru.md); устройство изнутри — в
[Для разработчиков](architecture.ru.md). Общий обзор — в [README](../README.ru.md).

---

## Требования к хостингу

| Компонент      | Требование                                                                                     |
|----------------|------------------------------------------------------------------------------------------------|
| PHP            | **8.1+** (разработка на 8.2)                                                                   |
| Расширения PHP | `pdo_mysql`, `curl`, `json` (обязательно); `mbstring` (желательно)                             |
| СУБД           | MySQL 5.7+ **или** MariaDB 10.x (InnoDB, utf8mb4)                                              |
| Веб-сервер     | Apache (`.htaccess` в комплекте) или Nginx (каталоги закрыть вручную) + `mod_rewrite`          |
| HTTPS          | **Обязателен** — вебхук и Login Widget работают только по HTTPS                                |
| Cron           | Возможность запускать `cron.php` по расписанию (раз в минуту — идеально; реже — тоже работает) |

`fastcgi_finish_request` **не требуется**: вебхук асинхронный (кладёт апдейт в очередь и
сразу отдаёт `200`, а очередь разбирает cron-воркер — см.
[Асинхронный вебхук](architecture.ru.md#асинхронный-вебхук)).

> **Единый источник времени — база данных.** Все метки времени хранятся и сравниваются в UTC
> по часам сервера БД (читаются один раз на соединение). PHP и БД могут стоять на разных
> хостах с разошедшимися часами — на корректность это не влияет.

Размещение задаётся через `APP_URL` и поддерживает оба варианта:
- поддомен: `https://ostrakon.example.com`
- подпапка: `https://example.com/ostrakon`

### Заметки по безопасности

- **`WEBHOOK_SECRET`** должен быть длинной случайной строкой: знающий его может слать поддельные
  апдейты в `webhook.php` (фейковые голосования/баны). Можно дополнительно ограничить `webhook.php`
  диапазонами IP Telegram на уровне веб-сервера.
- **Удалите `install.php`, `log.php`, `inspect.php`** после развёртывания; `DEV_TOKEN` оставьте пустым.
- Держите `SUPERADMIN_TOKEN` вне URL; cron/воркер использует отдельный низкопривилегированный
  `WORKER_TOKEN`.
- На **shared-хостинге** укажите каталог PHP-сессий внутри проекта (`session.save_path` +
  `open_basedir`), чтобы соседние арендаторы не читали файлы сессий панели. `config/`, `src/`,
  `logs/` должны быть закрыты по HTTP — на Apache это делает встроенный `.htaccess`, на Nginx
  настройте сами.

## Настройка бота (@BotFather)

Задача хостера у @BotFather — создать бота, глобально его настроить и (если нужно) вручную
зарегистрировать вебхук. Добавлять бота в группы и раздавать ему права в них — забота владельцев
групп, а не хостера.

### 1. Создать бота
`/newbot` → задать имя и username, оканчивающийся на `bot`. BotFather выдаст **токен** →
в `config/bot.php` (`BOT_TOKEN`).

### 2. Настроить бота
- **Выключите privacy-режим:** `/setprivacy` → выбрать бота → **Disable**. Делайте это **всегда**
  и **до того, как бота начнут добавлять в группы**: заранее вы не знаете, какие владельцы захотят
  режим full, а при включённом privacy бот не видит обычные сообщения и не может считать
  активность/score. (Само инициирование голосования через `@упоминание` работает при любом privacy.)
- **Домен для Login Widget** (для панели): `/setdomain` → выбрать бота → хост из `APP_URL`
  (например `ostrakon.example.com`, только хост — без `https://` и пути).
- **Аватар бота** (необязательно): `/setuserpic` → выбрать бота → загрузить квадратную
  картинку (Telegram обрежет её в круг). Можно взять `assets/favicon.png` (512×512). Там же
  задаются имя, `/setdescription` и `/setabouttext`.

### 3. Webhook
Установщик регистрирует вебхук автоматически из `APP_URL` + `WEBHOOK_SECRET`. Вручную это нужно
только если автоматическая регистрация не подошла:

```bash
curl -F "url=https://ВАШ_ДОМЕН/webhook.php" \
     -F "secret_token=ВАШ_WEBHOOK_SECRET" \
     -F 'allowed_updates=["message","callback_query","chat_member","my_chat_member"]' \
     "https://api.telegram.org/botВАШ_ТОКЕН/setWebhook"
```

> `allowed_updates` обязан включать `chat_member` и `my_chat_member`, иначе бот не увидит
> входы/выходы/повышения. Установщик передаёт правильный набор.

## Развёртывание

1. **Залить** проект в каталог, соответствующий `APP_URL`.
2. **Создать** БД (utf8mb4) и пользователя.
3. **Создать конфиги** из шаблонов и заполнить:
   - `cp config/bot.example.php config/bot.php` — `BOT_TOKEN`, `BOT_USERNAME`,
     `WEBHOOK_SECRET`, `APP_URL`, `LOG_LEVEL`. `SUPERADMIN_TOKEN` и `SUPERADMIN_PATH` оставить
     пустыми — их задаст первый запуск установщика (см. ниже).
   - `cp config/db.example.php config/db.php` — доступ к БД, `DB_TABLE_PREFIX`.
4. **Настроить бота** у @BotFather (шаги 1–2 выше: создать и настроить).
5. **Запустить установщик** — он проверит требования, применит **миграции**, зарегистрирует вебхук
   и меню команд бота в личных сообщениях (задавать команды вручную в @BotFather не нужно):
   - Веб, первый запуск (пустой `SUPERADMIN_TOKEN`): откройте `https://ВАШ_ДОМЕН/install.php` —
     токен не нужен; форма создаст логин/пароль супер-админа и запишет конфиг, затем установка пойдёт.
   - Веб, последующие запуски: `https://ВАШ_ДОМЕН/install.php?token=ВАШ_SUPERADMIN_TOKEN`
   - CLI: `php install.php`
   - Пропустить регистрацию вебхука: `?webhook=0` (веб) / `php install.php nowebhook` (CLI).
6. **⚠️ Удалить `install.php`** после успешной установки.
7. **Настроить системный cron** на `cron.php` (идеально — раз в минуту):
   - CLI: `* * * * * /usr/bin/php /полный/путь/к/cron.php >/dev/null 2>&1`
   - или по URL: `https://ВАШ_ДОМЕН/cron.php?token=ВАШ_WORKER_TOKEN` (для URL-cron сервиса,
     который не умеет слать заголовки; здесь намеренно используется низкопривилегированный
     `WORKER_TOKEN`, а не `SUPERADMIN_TOKEN`)
8. Убедиться, что `logs/` доступен веб-серверу на запись.
9. **Удалить dev-инструменты** `log.php` и `inspect.php` в проде (они только для отладки).
   Как страховка они также выключены, пока в `config/bot.php` не задан `DEV_TOKEN` (пуст по умолчанию).

Установщик идемпотентен (`CREATE TABLE IF NOT EXISTS`, `INSERT IGNORE`, учёт миграций) —
повторный запуск безопасен.

## Конфигурация

### `config/bot.php`
| Ключ               | Назначение                                                                              |
|--------------------|-----------------------------------------------------------------------------------------|
| `APP_URL`          | Полный публичный адрес (хост+путь, без хвостового `/`); единый источник всех ссылок     |
| `BOT_TOKEN`        | Токен от @BotFather                                                                     |
| `BOT_USERNAME`     | Username бота без `@` (Login Widget + распознавание упоминаний)                         |
| `WEBHOOK_SECRET`   | Проверяется в заголовке `X-Telegram-Bot-Api-Secret-Token` на каждом апдейте. **Возьмите длинное случайное значение** — знающий его может подделывать апдейты (фейковые голосования/баны) |
| `WORKER_TOKEN`     | Низкопривилегированный секрет, который ТОЛЬКО запускает воркер (`cron.php`). Self-пинок вебхука шлёт его в заголовке `X-Ostrakon-Token`; URL-cron передаёт как `?token=`. Отделён от `SUPERADMIN_TOKEN`, чтобы cron-URL в access-логах не выдал мастер-секрет. Генерируется на первом запуске `install.php` |
| `SUPERADMIN_TOKEN` | `base64("логин:пароль")`. Гейт для `install.php`/`migrations/run.php` по HTTP, для dev-инструментов **и** креды HTTP Basic Auth для страницы супер-админа. Держите его ВНЕ URL. Задаётся на первом запуске `install.php` |
| `SUPERADMIN_PATH`  | Секретный слаг страницы супер-админа → `APP_URL/<слаг>`; пусто — выключено (см. «Зона оператора» ниже). **Не должен содержать `admin`** — такой слаг конфликтует с панелью и игнорируется |
| `DEV_TOKEN`        | Отдельный секрет для dev-эндпоинтов `log.php`/`inspect.php` (не `SUPERADMIN_TOKEN`). **Пусто по умолчанию → эндпоинты выключены** (отдают `404`, даже если файлы остались). Задавайте случайное значение только на время отладки |
| `LOG_LEVEL`        | `trace`/`debug`/`info`/`warning`/`error`/`fatal`                                        |

### `config/db.php`
Параметры подключения и `DB_TABLE_PREFIX` (напр. `vb_`).

### `config/defaults.php`
Дефолты групп (копируются в `groups` при добавлении бота), настройки инстанса
(`history_days`, тайминг воркера) и интервалы cron. Подробности — в комментариях файла.

## Зона оператора (супер-админа)

Если вы держите бота для других, есть отдельная страница оператора: все группы, где работает
бот — включая те, откуда его удалили, — с последней активностью каждой, а для выбранной группы —
её владелец и админы (боты исключены). Нужна, чтобы напрямую связаться с владельцами/админами
групп, например предупредить перед закрытием хостинга. Оператору **не нужно ничего знать про
Telegram**: данные о группах сервер берёт по токену бота, а сама страница к Telegram-логину
отношения не имеет.

Страница живёт по **секретному адресу, который вы задаёте** — ключ `SUPERADMIN_PATH` в
`config/bot.php` (случайный слаг → `APP_URL/<слаг>`) — и закрыта **HTTP Basic Auth**
(`SUPERADMIN_TOKEN`, хранится как `base64("логин:пароль")`). Два независимых слоя: секретный URL и
пароль. Оставьте `SUPERADMIN_PATH` пустым — и страница выключена (любой запрос → обычный `404`).

Креды задаются на **первом запуске `install.php`**: пока `SUPERADMIN_TOKEN` пуст, установщик не
требует токена и показывает короткую форму создания логина/пароля — он сам впишет `SUPERADMIN_TOKEN`
и случайный `SUPERADMIN_PATH` в `config/bot.php` (сделайте файл доступным на запись — или вставьте
показанные значения вручную). Если есть shell, `php install.php gentoken` печатает те же значения
для вставки. На CGI/FastCGI-хостинге для Basic Auth нужен проброс заголовка `Authorization` к PHP —
это уже делает `.htaccess` из комплекта (альтернатива для Apache — `CGIPassAuth On`).

## Логирование и dev-инструменты

`logs/app.log` (время в UTC). Уровень — `LOG_LEVEL`:
`FATAL(60) > ERROR(50) > WARNING(40) > INFO(30) > DEBUG(20) > TRACE(10)` — пишутся
сообщения важностью ≥ порога. При отладке — `debug`/`trace`, в проде — `warning`/`error`.

Только для разработки — **удалите в проде**: `log.php` (просмотр/очистка лога) и
`inspect.php` (инспектор/правилка БД — произвольный SQL). Закрыты отдельным `DEV_TOKEN`
(не `SUPERADMIN_TOKEN`) и **выключены, пока `DEV_TOKEN` не задан** — по умолчанию пуст, поэтому
отдают `404`, даже если файлы на месте. Первый заход с `?token=DEV_TOKEN` ставит cookie.
CLI работает всегда.
